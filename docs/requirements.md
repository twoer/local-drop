# LocalDrop 需求文档

> 版本：v1.3  
> 创建日期：2026-04-29  
> 最后更新：2026-04-29（二次审查：clientId 持久化统一 / 单远程房间限制 / OS 兜底）  
> 状态：草案

## 1. 项目概述

### 1.1 一句话描述

一个基于浏览器的设备互传工具，支持同局域网设备自动发现、外网通过配对码加入，通过 WebRTC P2P 互发文字、图片、视频、压缩包等内容。

### 1.2 解决的问题

- 家里 Mac、Windows、手机之间需要快速互传文件/链接/文字
- AirDrop 不跨平台、微信文件传输助手压缩画质且有大小限制
- 想要一个无需安装、打开网页就能用的零门槛工具

### 1.3 灵感来源

- [SnapDrop](https://github.com/RobinLinus/snapdrop) / [PairDrop](https://github.com/schlagmichdoch/PairDrop)：架构参考
- AirDrop：交互参考

---

## 2. 使用场景

### 2.1 主场景（家庭 LAN，~90%）

- 同 WiFi 下的 Mac、Windows、手机互传
- 自动发现设备，自动接收文件，无需任何确认操作
- 典型操作：从手机发图到 Mac、从 Mac 发压缩包到 Windows

### 2.2 次场景（跨网络，~10%）

- 出门在外，需要给家里的设备发个文件 / 给同事的设备发个东西
- 双方都不在同一 LAN，通过公网信令服务 + 4 位配对码加入同一房间
- 文件较小、需要确认接收

### 2.3 不在 v1 范围内的场景

- 多人协作 / 团队工作空间
- 公开链接分享（生成 URL 给陌生人下载）
- 大文件传输（GB 级别，由次要场景隐含限定）

## 3. 用户与设备

### 3.1 目标设备

| 设备类型 | 优先级 | 说明 |
|---------|-------|------|
| macOS（Chrome/Safari） | P0 | 必须支持 |
| Windows（Chrome/Edge） | P0 | 必须支持 |
| 手机浏览器（iOS Safari、Android Chrome） | P0 | 必须支持 |
| Linux 桌面 | P1 | 不主动测，但应该可用 |

### 3.2 浏览器要求

- 必须支持 WebRTC（DataChannel）
- 必须支持 IndexedDB
- 必须支持 WebSocket
- 必须为 HTTPS 上下文（localhost 除外）

### 3.3 用户角色

- 单一角色：**普通用户**
- 不区分管理员、不需要登录、不需要注册

### 3.4 并发规模

- 单房间内同时在线 **3+ 台设备**
- 全站并发：家庭使用，预计 < 10 个房间同时活跃
- v1 信令服务无需引入 Redis，内存存储房间状态即可

## 4. 功能需求

### 4.1 设备发现与配对

#### 4.1.0 信令与数据的关系（重要前提）

**所有模式下，文件和消息内容都走 WebRTC P2P，从不经过服务器**。信令服务器只负责"撮合"——交换 SDP/ICE 这几 KB 的握手数据，看不到也存不到任何用户内容。

> ⚠️ 浏览器无法独立完成 WebRTC 信令交换（不能监听端口、不能广播 mDNS），所以**任何模式都需要信令服务器在线**。"不走服务器"指的是数据流，不是信令流。  
> 真正完全脱离服务器的方案（QR 码扫码交换 SDP）见 §9 后续版本规划。

#### 4.1.1 入口模式选择

进入页面时，用户首先选择一种使用模式（也可由 URL 参数预设跳过此步）：

| 模式 | 适用场景 | UI 表现 |
|------|---------|--------|
| 🏠 **局域网模式** | 家里 / 同 WiFi | 进入后自动发现同 LAN 设备，**完全隐藏配对码 UI** |
| 🌐 **跨网模式** | 出门在外 / 跨网络 | 显示"创建房间 / 加入房间"两个入口 |
| 🔀 **智能模式（默认）** | 不确定时 | LAN 自动发现 + 同时允许通过配对码扩展加入 |

模式选择会记忆在 localStorage，下次进入直接使用上次的选择，提供"切换模式"入口。

**URL 参数支持**（用于分享链接 / 快捷进入）：

| URL | 行为 |
|-----|------|
| `/` | 入口页（首次访问 / 已退出） |
| `/?mode=lan` | 跳过入口页，直接进入局域网模式 |
| `/?mode=wan` | 跳过入口页，直接进入跨网模式 |
| `/?mode=smart` | 跳过入口页，直接进入智能模式 |
| `/?join=1234` | 跨网模式 + 自动加入指定房间（适合分享给对方） |

优先级：**URL 参数 > localStorage 记忆 > 显示入口页**。所有路由统一在 `/`，由 query string 决定行为。

#### 4.1.2 局域网自动配对（局域网模式 / 智能模式）

- 用户进入页面，无需任何操作
- 信令服务器从 WebSocket 连接的 remote address 取得**公网出口 IP**
- 判定规则极简：**公网出口 IP 相同即视为同一"广义 LAN"**

**为什么不做更细的内网网段比对**：

现代浏览器（Chrome 2019+、Safari、Firefox）出于隐私默认对 WebRTC ICE host 候选地址做 **mDNS 屏蔽**——服务端拿到的不是真实 `192.168.x.x`，而是形如 `xxx.local` 的随机主机名，**无法用于子网比对**。除非请求 `getUserMedia` 权限解锁（UX 不可接受），否则没有可靠的服务端方式获取真实内网 IP。

因此 v1 简化为：
- **同公网 IP = 同广义 LAN**（覆盖单路由双频、AP 透传、Mesh 透传等所有家庭网络拓扑）
- 双频段 / 主路由 + 路由模式子路由（不同 `/24`）→ **仍视为同 LAN**（原本要担心的"误判"反而成了优势）

**误判与兜底**：

| 场景 | 处理 |
|------|------|
| 运营商级 NAT（CGNAT）下不同家庭误判同网 | 用户在设备列表中点 **"我不在这个网络里"** 按钮主动退出 LAN 自动房间 |
| 同公网 IP 但跨 VPN / VLAN 想分隔 | v1 不区分；用户可改用配对码模式或上述按钮 |

- 同 LAN 设备自动加入同一个"自动房间"，互相可见
- 客户端可手动退出 LAN 房间（不影响配对码房间）

#### 4.1.3 配对码房间（跨网模式 / 智能模式）

提供两个入口：

- **创建房间**：系统生成一个 4 位**纯数字**配对码（`0000`–`9999`），页面展示，等待其他设备加入
- **加入房间**：输入框让用户输入对方的 4 位配对码（仅数字键盘），加入对应房间

配对码规则：
- 长度 4 位，**纯数字**（共 10000 组合）
- 创建后立即生效，**直到创建者关闭页面**为止
- 同一时刻全站不重复（信令服务保证）
- 房间内成员关系自动维护（断线重连后能恢复）

**冲突与异常处理**：

| 场景 | 处理 |
|------|-----|
| 创建时生成的码已被占用 | 后端**自动重试**生成新的，用户**无感知**，看到的就是最终可用的码 |
| 极端：码池耗尽（10000 全占满） | 创建失败 → Toast："服务器繁忙，请稍后再试" |
| 加入时输入的码不存在 / 已销毁 | 输入框下方红色提示："**房间不存在或已关闭**"，输入内容保留方便修改 |
| 加入时输入格式错误（非 4 位数字） | 实时校验，禁用提交按钮 |

> ⚠️ 4 位纯数字仅 10000 组合，后续如出现房间数量逼近上限的情况，需要升级为更长或字母数字混合（v1 家用场景足够）。

> 边界：用户在外网创建房间后，家里的设备如果也想加入，同样需要输入配对码——v1 不做"我自己的设备自动跨网识别"。

#### 4.1.4 智能模式行为

- 进入后**先尝试 LAN 自动发现**（同 §4.1.2）
- UI 上同时提供"创建房间 / 加入房间"按钮
- 用户可同时存在于多个独立的房间中（最多 3 个：1 LAN + 1 自创配对码 + 1 加入的远程配对码）

**v1 房间数量限制**：
- 同时最多 1 个 LAN 自动房间（由公网 IP 决定，自然只有一个）
- 同时最多 1 个**自己创建**的配对码房间
- 同时最多 1 个**已加入**的远程配对码房间（再加入新的会先离开旧的）

> 限制理由：避免 mesh 拓扑随房间数量爆炸（多房间 × 多 peer），简化 UI 状态管理。v2 视用户反馈再放开。

**关键：两类房间是隔离的（避免隐私泄漏）**

| 房间 | 成员 |
|------|------|
| LAN 自动房间 | 我 + 同 LAN 的其他设备 |
| 我创建的配对码房间（"我的个人房间"） | 我 + 通过配对码加入的远程设备 |

- 远程设备 D 通过配对码加入的是**我的个人房间**，**不会看到**我所在的 LAN 房间里的其他设备（A、B、C）
- A、B、C 也**不会看到** D（除非 A/B/C 自己也输入了同一个配对码）
- UI 上消息流可合并展示（按时间线），但每条消息的"目标设备"只会是当前房间的成员
- 设备列表用网络位置标签区分：🏠 LAN 设备 / 🌐 配对码房间设备

#### 4.1.5 P2P 拓扑：Mesh（全连接）

房间内 N 台设备两两建立 PeerConnection，共 `N×(N-1)/2` 条连接。

```
3 台设备：3 条连接
4 台设备：6 条连接
5 台设备：10 条连接
```

**为什么选 Mesh**：
- 群聊式语义最自然（每对独立通道）
- 无单点故障（任一设备掉线不影响其他对）
- 不需要选主 / 重选主算法
- 无中转，延迟最低
- 对于 v1 目标规模（≤6 台 / 房间，文件 ≤10MB）资源开销可接受

**发送行为**：
- 文字消息：广播给当前房间内所有 peer（每条 PeerConnection 各发一次）
- 文件：同上，发送方上行流量 = 文件大小 × (N-1)
- v1 不做 gossip 转发优化（v2 视需要再加）

**成员动态**：
- 新成员加入 → 与现有所有成员两两建立 PeerConnection
- 成员离开 → 关闭涉及该成员的所有 PeerConnection，其他连接不受影响

#### 4.1.6 设备命名与识别

- **不支持用户改名**（v1）
- 设备名 = `{操作系统} + {浏览器}`，由前端从 UA 解析后传给信令服务器
  - 示例：`Windows + Edge`、`macOS + Chrome`、`iOS + Safari`、`Android + Chrome`
  - 不在白名单的 OS（如 ChromeOS、BlackBerry 等）→ 显示为 `Unknown`
  - UA 完全解析失败 → 显示为 `Unknown + Unknown`，不阻塞使用
- 同型号设备区分：同一房间出现两台 `macOS + Chrome` 时，自动追加序号 `macOS + Chrome (2)`
- 客户端持久化：每个浏览器 tab 首次进入时生成 UUID 形式的 `clientId`，存 **sessionStorage**（每 tab 独立），用于刷新重连识别（见 §4.4）。详细原因见设计文档 ADR-008

### 4.2 消息与文件传输

#### 4.2.1 支持的内容类型（按优先级）

| 优先级 | 类型 | 备注 |
|-------|-----|------|
| P0 | 文字消息 | 含 URL、长文本，支持复制 |
| P0 | 图片 | jpg/png/webp/gif，可预览 |
| P0 | 视频 | mp4/mov/webm 等 |
| P0 | 压缩包 | zip/rar/7z 等任意二进制 |
| P1 | 任意其他文件 | 一并按二进制处理 |

> 实现上文字与文件**走同一条 WebRTC DataChannel**，按消息类型分发即可。

#### 4.2.2 文件大小限制

- **99% 场景：单文件 ≤ 10 MB**
- 偶发场景：可能超过 10 MB，不强制限制
- v1 不限定上限，但 UI 上对 > 50 MB 文件给出温和提示（"较大文件可能传输时间较长"）

#### 4.2.3 接收行为与渲染策略

文件到达后**不自动触发浏览器下载**，而是根据类型采用不同的展示方式：

| 类型 | 展示方式 | 用户操作 |
|------|---------|--------|
| 文字 | 直接以气泡形式显示 | 可复制 |
| 图片 (`image/*`) | 用 `<img>` 直接渲染缩略图 | 点击放大预览 / 长按或右键保存 |
| 视频 (`video/*`) | 用 `<video controls>` 直接渲染播放器 | 在线播放 / 下载 |
| 音频 (`audio/*`) | 用 `<audio controls>` 渲染 | 在线播放 / 下载 |
| 压缩包 / PDF / 其他 | 卡片形式：图标 + 文件名 + 大小 + **下载按钮** | 点击下载按钮才触发浏览器下载 |

> 设计原则：常用媒体（图片/视频）即开即看，**不污染下载目录**；非媒体类文件用户主动选择是否下载到本地。

**所有平台一致**：不再区分"局域网自动下载 / 跨网手动接收"——统一为"自动展示，按需下载"。

**移动端注意**：
- iOS Safari 的"下载"行为是触发系统分享菜单（保存到照片/文件 App）
- Android Chrome 是直接保存到下载目录

#### 4.2.4 多设备交互模型

- **群聊式**：房间内所有设备共享同一个消息流
- 任何一台设备发消息/文件，房间内其他所有设备都能收到
- v1 **不支持** 1 对 1 私发

### 4.3 历史记录

- 使用 **IndexedDB** 在本地持久化消息和文件
- 刷新页面、关闭浏览器后能看到历史
- 文件以 Blob 形式存储在 IndexedDB 中，可重新下载
- 提供"清空历史"按钮（v1 全量清空，不做按时间筛选）
- 历史记录与房间无强绑定，本机就是真相源

### 4.4 房间生命周期与连接保持

#### 4.4.1 房间生命周期

- **自动 LAN 房间**：根据网络指纹动态归属，不需要手动管理；无成员后立即销毁
- **配对码房间**：
  - 创建者**关闭页面**才销毁（关闭页面 = WebSocket 关闭 + 宽限期超时未恢复）
  - 创建者**刷新页面**不销毁（见下文）
  - 创建者关闭后，房间内其他成员被踢出，UI 提示"房间已关闭"
- 信令服务在房间销毁后立即释放配对码，重新进入码池

#### 4.4.2 刷新与短暂断线的保持机制

设计目标：**刷新页面 / 网络抖动 5 秒内重连**应保持房间身份不丢失。

实现：
- 客户端维护一个持久化的 `clientId`（UUID，存 **sessionStorage**，每 tab 独立）
- 客户端连接信令服务时携带 `clientId` + 当前房间标识
- 服务端按 `clientId` 维护成员身份：
  - WebSocket 断开 → **不立即移除**该成员，进入 **15 秒宽限期**
  - 宽限期内同 `clientId` 重连 → 直接恢复其在房间中的身份
  - 宽限期超时仍未重连 → 才移除成员，通知其他设备
- 创建者重连后房间继续有效；其他成员重连后无需重新输入配对码

> 宽限期 15 秒覆盖：刷新页面（约 1-3 秒）、WiFi 抖动（约 5 秒）、4G 切换（约 5-10 秒）。

#### 4.4.3 WebRTC 连接的恢复

- 信令层重连成功后，自动重新协商失效的 PeerConnection
- 不强求"WebRTC 上下文也保留"——重新建连即可，由于内容已在 IndexedDB，体验上无感

## 5. 非功能需求

### 5.1 性能

- WebRTC 建连完成后，10 MB 文件局域网传输应在 **5 秒内**完成
- 信令握手时间：从打开页面到能看到同 LAN 设备 < **3 秒**

### 5.2 安全

- 所有页面强制 HTTPS（已具备证书）
- WebRTC 自带 DTLS 加密，**v1 不做应用层 E2EE**
- 信令服务器不存储任何文件内容（只转发信令包）
- 配对码只在内存中存在，无持久化

### 5.3 可用性

- 进入页面到能用：**3 秒内**
- 移动端友好：响应式布局，触屏交互
- 拖拽上传 + 粘贴上传 + 点击选择：三种方式都支持

### 5.4 可观测性

- 服务端记录信令事件日志（房间创建/销毁、配对码生成、连接错误）
- 不记录消息/文件内容
- 客户端连接状态对用户可见（"已连接 / 连接中 / 离线"）

### 5.5 错误场景与处理策略

按错误的影响范围分四类。

#### 5.5.1 阻断型（全屏错误页）

| 场景 | 提示 |
|------|-----|
| 浏览器不支持 WebRTC | "请使用最新版 Chrome / Edge / Safari / Firefox" |
| 浏览器不支持 IndexedDB | 同上 |
| 非 HTTPS 上下文（且非 localhost） | "请通过 HTTPS 访问本应用" |

#### 5.5.2 信令层错误（Toast + 自动重试）

| 场景 | 处理 |
|------|-----|
| WebSocket 首次连不上 | 指数退避重连：1s → 2s → 4s → 8s → 16s，3 次失败后 Toast "信令服务暂时不可用" |
| WebSocket 中途断开 | 静默自动重连，UI 显示"连接中" |
| 创建房间失败（码池满） | Toast "服务器繁忙，请稍后再试" |
| 加入码不存在 / 已销毁 | 输入框下红色提示"房间不存在或已关闭" |

#### 5.5.3 P2P 层错误（设备级提示）

| 场景 | 处理 |
|------|-----|
| 与某 peer 建连失败（强 NAT / 防火墙） | 该设备显示"⚠️ 无法直连"，给出排查建议链接 |
| 已连 peer 中途掉线 | 设备状态变灰，自动重连一次，仍失败则从列表移除 |
| 文件传输中断 | 该消息卡片显示"传输失败" + 重试按钮 |

#### 5.5.4 资源层错误

| 场景 | 处理 |
|------|-----|
| IndexedDB 写入失败（容量不足等） | Toast "本地存储已满，无法保存历史记录" |
| 文件渲染失败（损坏图片 / 不支持格式） | 降级为图标卡片 + 下载按钮 |

#### 5.5.5 v1 不做

- 用户主动取消传输（v2）
- 接收方拒绝接收（v2，配合容量管理）

## 6. 技术选型

| 层 | 选型 |
|---|------|
| 框架 | Nuxt 4（前后端一体） |
| UI | Vue 3 + TypeScript |
| 组件库 | shadcn-vue |
| 样式 | Tailwind CSS |
| 状态管理 | Pinia |
| 服务端 | Nuxt Nitro + WebSocket（`crossws`） |
| 实时传输 | 浏览器原生 WebRTC（DataChannel） |
| 本地存储 | IndexedDB（建议封装层 `idb` 或 `dexie`） |
| 部署 | 自有服务器 + 自有 HTTPS 证书 |

### 6.1 不引入的依赖（v1）

- Redis（房间状态用内存即可）
- 数据库（无账号无持久化业务数据）
- TURN 服务器（v1 仅支持能 P2P 直连的网络，不能直连的提示用户）

> ⚠️ 风险点：双方都在严格 NAT 后时（如运营商级 NAT），WebRTC 可能打洞失败。v1 接受这个限制，用户会看到错误提示。后续版本再考虑加 TURN。

## 7. 页面结构

### 7.1 入口页（`/` 无参数 / 无记忆时）

三选一卡片：

- 🏠 **局域网模式** — 适合家里 / 同 WiFi
- 🌐 **跨网模式** — 适合跨网络传输
- 🔀 **智能模式（推荐）** — 自动适配

选择后切换到主页视图，并在 localStorage 记忆当前模式。提供"下次直接进入"勾选项；已记忆或带 URL 参数（`?mode=...` / `?join=...`）时启动跳过此页。

### 7.2 主页（`/` + 模式参数 / localStorage 记忆）

顶部状态栏：
- 当前模式标识（🏠 / 🌐 / 🔀）+ "切换模式"按钮
- 网络状态指示器（已连接设备数）

模式相关区域（按所选模式展示）：
- **局域网模式**：仅显示自动发现的同 LAN 设备列表
- **跨网模式**：显示"创建房间 / 加入房间"按钮 + 房间内设备列表
- **智能模式**：同 LAN 设备 + "创建/加入跨网房间"入口并列

通用区域（所有模式共享）：
- 设备列表（设备名 + 平台图标 + 网络位置标签 🏠/🌐）
- 消息流（群聊式时间线）
  - 文字气泡
  - 图片缩略图（可点击预览）
  - 视频播放器
  - 文件卡片（图标 + 文件名 + 大小 + 下载按钮）
- 输入区
  - 文字输入框（Enter 发送）
  - 文件按钮 / 拖拽区域 / Ctrl+V 粘贴
- 清空历史按钮

### 7.3 暂不需要的页面

- 设置页（v1 全部默认，无可配置项）
- 关于页（README 即可）
- 错误页（用 Toast 替代）

## 8. 边界与约定

### 8.1 明确不做

- ❌ 用户账号系统
- ❌ 跨设备消息云同步（你的历史只在本机）
- ❌ 私聊/分组
- ❌ 文件断点续传（v1 简化）
- ❌ 离线消息（双方必须同时在线）
- ❌ PWA / 安装到桌面
- ❌ 多语言（v1 中文为主，可后期加）
- ❌ TURN 中继（强 NAT 下不保证可用）
- ❌ 用户改设备名（自动 OS+浏览器 命名）
- ❌ 消息状态语义（已发送 / 已送达 / 已读，v1 不做）
- ❌ IndexedDB 容量管理与自动清理（v2）
- ❌ 文件去重（同一文件发给多设备时本地多份，v2）
- ❌ Mesh 拓扑的 gossip 转发优化（v2 视情况）
- ❌ Markdown 渲染（文字消息一律按纯文本展示，URL 自动可点击；v1.x 再加）

### 8.2 明确做

- ✅ 三种入口模式选择（局域网 / 跨网 / 智能）
- ✅ LAN 零配置自动发现
- ✅ WAN 4 位配对码加入
- ✅ 群聊式多设备同步
- ✅ IndexedDB 本地历史
- ✅ 文字/图片/视频/任意文件
- ✅ 移动端响应式
